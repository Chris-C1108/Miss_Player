# 开发日记 — 2026年7月16日
## iOS Safari 滚动橡皮筋拦截与全局滚动穿透控制

---

## 1. 核心需求与实现功能

### 1.1 全局触摸与选择隔离 (Touch & Selection Isolation)
- **非穿透组件的绝对隔离**：除视频区域上滑穿透滚动底层网页外，播放器其他所有组件（控制面板、展开抽屉、按钮区等）必须完全隔离，禁止触摸和鼠标穿透操作底层原始网页，同时禁用文本选择和长按气泡菜单。
- **CSS 第一防线配置**：
  - 在 `.tm-video-overlay`、`.tm-bottom-sheet-overlay` 等全局遮罩上应用 `touch-action: none;`，彻底锁死底层穿透。
  - 在 `.tm-player-container`、`.tm-video-overlay`、`.tm-bottom-sheet-overlay` 上声明 `user-select: none; -webkit-user-select: none; -webkit-touch-callout: none;`，全面禁用文本高亮及 iOS 独有的长按图片/链接弹出菜单。
  - 针对非视频的非滚动子元素（如 `.tm-control-buttons`, `.tm-handle-container`, `.tm-button-container`, `.tm-settings-panel`），显式设置 `touch-action: none;` 以免发生误触穿透。

### 1.2 视频画面区域原生滚动穿透 (Native Scroll Penetration)
- **突破 touch-action 树交集限制**：
  - CSS `touch-action` 的生效取决于事件目标到可滚动祖先路径上所有元素 touch-action 值的交集。
  - **修复方案**：将根容器 `.tm-player-container` 的 `touch-action` 从 `none` 改为 `pan-y`，并把最底层的事件目标 **`<video>` 元素本身的 `touch-action` 从 `none` 更改为 `pan-y`**。这为垂直平移手势打开了合法的浏览器物理通道。
- **解除 pointerdown / touchstart 首帧阻止默认行为拦截**：
  - **问题成因**：原手势引擎 `videoSwipeManager.js` 的 `_handlePointerDown`（即首帧）无条件调用了 `e.preventDefault()`。在移动端，首帧被 preventDefault 会使浏览器直接废弃该手势在当前生命周期内的所有原生动作（包括页面滚动），从而导致穿透失效。
  - **解决方案**：注释掉 `_handlePointerDown` 中的 `e.preventDefault()`，仅在 `_handlePointerMove` 的 `touchmove` / `pointermove` 阶段，对非上滑方向（如左右滑动 seek、向下滑动折叠面板）进行方向判断并选择性调用 `preventDefault()`，对于上滑方向则直接放行，成功触发浏览器原生滚动冒泡。
- **解决 WebKit 时序差**：
  - 针对 Safari 中 `touchmove` 事件比 `pointermove` 先一步触发的时序差，在 `_touchPreventDefault` 拦截器中不再读取异步的 `this.deltaY`，而是直接通过 `e.touches[0].clientY` 与 `this.startY` 进行同步计算，精准判断滑动方向。

### 1.3 宿主网页橡皮筋弹性回弹拦截 (Disable Viewport Rubber-Banding)
- **根源机制控制**：在播放器激活时，向 `document.body` 和 `document.documentElement` 添加激活状态类名 `tm-player-active`，并动态注入全局样式，对 `html` 和 `body` 双层结构应用 `overscroll-behavior: none !important;`，切断 Safari 在页面边缘的橡皮筋回弹以及下拉刷新，确保固定定位布局的播放器整体画面不会发生物理位移与跳跃。

## 2. 调试与诊断手段：屏幕悬浮调试日志窗口 (DebugLogger)

为了在 iOS Safari 等移动端（难联调 Web Inspector 的环境）进行高效排查，本版本引入了**屏幕浮动调试日志窗口**：
- **实时事件追踪**：实时捕获并滚动输出 touchstart、touchmove、window scroll 的 ClientY 坐标、滑动 delta 差值和 window.scrollY 滚动位置。
- **一键拷贝设计**：集成 `Copy` 按钮，并在移动端环境下通过 fallback textarea 机制，确保长按一键拷贝日志，使用户可以轻松将日志数据反馈给开发端。
- **排错功臣**：通过日志发现 `touchmove video [SWIPE UP]. diffY: -123. Allow!` 正常输出而宿主 `scrollY: 277` 始终不变，从而精准定位并攻克了 `<video>` 本身 `touch-action: none` 的硬件手势阻断机制。

## 3. 踩坑教训与开发经验

### 3.1 移动端 touch-action 的生效机制
- **交集原则**：一个手势是否被浏览器允许，取决于整个 DOM 事件路径上所有节点的 `touch-action` 值的交集。如果祖先节点中有一个是 `none`，或者最底层叶子节点（如 `video` 标签）是 `none`，那么任何平移都无法唤醒浏览器原生滚动。
- **最佳实践**：在需要穿透的事件路径上（如 `video` -> `video-wrapper` -> `video-container` -> `player-container`），必须全线放行 `pan-y`，并在非穿透的同级子元素上显式设置为 `touch-action: none` 来实现定向物理隔离。

### 3.2 阻止默认行为 (preventDefault) 的生命周期阻断
- **首帧敏感**：在 `touchstart` / `pointerdown` 阶段调用 `preventDefault()` 会直接废弃该手势在当前生命周期的原生滚动机会。若需要支持原生穿透，必须确保首帧不被阻止，而在随后的 `touchmove` / `pointermove` 阶段进行方向性条件阻止。

## 4. 水平滚动容器的触摸滑动隔离修正

### 4.1 标签页栏与评论源选项卡垂直滑动穿透问题
- **成因**：由于根容器 `.tm-player-container` 改为了 `touch-action: pan-y` 以放行视频区上滑穿透，且水平滚动容器（`.tm-tab-scroll-container` 与 `.tm-comments-tabs`）本身没有限定手势方向，导致在这些区域垂直滑动时，手势会继承或解析为 `pan-y`，且事件会直接穿透并滚动底层原始网页。
- **治理方案**：
  - **CSS 物理限制**：为 `.tm-tab-scroll-container`（标签页栏）与 `.tm-comments-tabs`（评论源选项卡）显式设置 **`touch-action: pan-x;`**。这在合成器层面限制了它们只允许水平滑动（scroll 横向标签），垂直平移手势会被物理丢弃。
  - **JS 逻辑卡点**：在 `EventManager.js` 中追踪 `_touchStartX` 和 `_touchStartY`。在 `touchmove` 阶段，如果检测到触碰发生在这些水平滚动容器内，计算出横纵位移并进行对比：
    - 若 **`diffY > diffX`**（即以垂直滑动为主的手势），则同步执行 **`e.preventDefault()`** 强行拦截，彻底杜绝任何垂直方向的手势外溢与穿透。
    - 若水平滑动占优势，则放行以支持正常的横向平移滚动。

## 5. iOS Safari/移动端自动进入系统全屏播放器治理

### 5.1 自动全屏问题成因
- 在 iOS Safari、微信内置浏览器等移动端环境下，如果 `<video>` 标签没有被明确指定内联播放属性，一旦视频开始播放（或宿主脚本触发 play 事件），系统会自动唤起 iOS 默认的全屏多媒体播放器（QuickTime 容器界面），将视频脱离出 Miss Player 的自定义控制骨架和 UI 界面。

### 5.2 治理与隔离方案
- **注入内联播放属性**：
  在 `UIManager.js` 中创建视频 wrapper 并接入原始视频元素时，无条件为该视频元素设置内联播放关键属性：
  - `playsinline="true"`
  - `webkit-playsinline="true"`
  - `x5-playsinline="true"`
  - 同时同步设置对应的 DOM 属性值 `playsInline = true` 与 `webkitPlaysInline = true`。
- **原始属性恢复**：
  在 `PlayerCore.js` 准备接管视频前，通过 `saveVideoState()` 将宿主原始视频的 `playsinline` 属性值保存至 `videoState`。在关闭播放器 `close()` 时，将原属性及 DOM 值恢复回原本的状态，做到对宿主页面完全无损。
- **触发播放时二次强化拦截**：
  在 `EventManager.js` 的 `handlePlayBound`（`play` 事件监听器）中，每次视频被播放时，均会重新为该 `targetVideo` 注入/强化这三个 playsinline 属性。这保证了即使宿主站点的其他脚本在播放时动态重建了视频节点或擦除了属性，也会被我们立刻覆盖回去，彻底杜绝系统原生播放器的自动弹出。

## 2. 踩坑教训与避坑经验

### 2.1 状态持久化键名不匹配导致的“全局设置失效”幽灵 Bug
* **坑点**：在重构预加载防漏逻辑时，在 `CommentPanel.preload` 中尝试使用 `GM_getValue('tm_player_settings')` 读取配置对象。然而底层 `PlayerState.js` 实际采用的是 `getValue('showCommentsSection', true)` 逐个键名独立存储的方式（在 GM 存储中键名为 `showCommentsSection` / `enabledCommentSources`，而不是一个统一的 JSON 对象 `tm_player_settings`）。这导致 `preload` 方法读到的永远是 `null`，后端的拦截守卫完全失效，控制台中依然看到脚本去采集所有站点的评论。
* **经验**：在非面向对象实例的静态方法或外部模块中读取存储状态时，**必须与系统全局存储架构（`PlayerState` / `storage.js`）的键名规范及存储模式保持 100% 严格一致**。涉及 GM 存储与 localStorage 混合降级时，应当优先封装或调用统一的存储工具函数，切忌凭主观猜测凭空硬编码键名。

### 2.2 多层级并发/预加载逻辑的“防御性配置校验”
* **坑点**：为了优化首屏加载速度，系统往往会在主程序初始化前（Boot Time）提前触发后台静默预加载（如 `CommentPanel.preload`）。但如果预加载逻辑独立于常规 UI 渲染流程之外，且没有在触发的最顶端校验用户持久化设置，就会导致用户明明在 UI 里禁用了某功能，后台网络请求依然大张旗鼓发出的怪异现象。
* **经验**：任何异步预加载、后台静默请求或 Hook 模块，在执行耗时/跨域网络操作的第一行代码处，都必须添加“防御性配置校验（Defensive Configuration Check）”，保证用户的个性化设置在系统的每一个角落（包括最前置的 Boot Preload）都能生效。

### 2.3 软键盘/触摸屏下的流式无缝横向滚动（No-wrap Flex Horizontal Scroll）
* **坑点**：在 Modal/Popover 等小尺寸弹窗内嵌入横向滚动条时，如果简单使用 `overflow-x: auto`，常常会因为 Flex 子元素的收缩机制（`flex-shrink: 1`）导致胶囊按钮被挤压变形，或者因为没有处理 `white-space` 而在小屏设备上发生换行破版。
* **经验**：实现干净的无缝流式胶囊横向滚动三要素：
  1. 父容器设置 `flex-wrap: nowrap; overflow-x: auto; white-space: nowrap; -webkit-overflow-scrolling: touch;`
  2. 隐藏原生粗糙滚动条 (`scrollbar-width: none; ::-webkit-scrollbar { display: none; }`)
  3. 子元素必须显示声明 **`flex-shrink: 0;`**，防止内部控件在空间不足时遭弹性挤压；同时针对桌面端绑定 `wheel` 事件将竖向滚轮位移映射至 `scrollLeft`。

### 2.4 Flex 容器下文本与胶囊按钮的像素级垂直居中
* **坑点**：设置 `height` 与 `line-height` 在不同浏览器或字体渲染引擎下，经常因字体 Metrics（Baseline / Ascender / Descender）的微小差异导致文本上下偏离 1~2px。
* **经验**：避免单纯依赖 `line-height` 进行垂直居中。对于按钮、徽章或胶囊组件，推荐使用 **`display: inline-flex; align-items: center; justify-content: center; line-height: 1;`**，配合显式 `height` 与 `padding`，能确保跨平台下的绝对像素级垂直居中。

#### 3. 画面左右平移 Minimap 微缩图按需淡入、渐隐与非标准比例溢出修复
* **彻底默认隐退与手势平移即时淡入**：
  * 将 `.tm-video-minimap` 的默认 CSS 状态设为 `opacity: 0 !important; visibility: hidden !important;`，在常态播放或静止时彻底隐藏，解决以往 Minimap 无脑常驻显示的问题。
  * 仅在检测到手势水平平移 (`.is-swiping`) 时，通过 CSS 淡入高亮显示 (`opacity: 1; visibility: visible; border-color: rgba(...)`)。
  * 手势平移结束释放后，JS 延迟 600ms 移除 `.is-swiping` 类，结合 `transition: opacity 0.3s ease, visibility 0.3s ease` 实现自然平滑的淡出隐退。
* **非标准宽高比 (1:1 / 9:16) 下的 Minimap 尺寸自适应与边界防溢出**：
  * 重构 JS 中 Minimap 动态宽高计算逻辑。以往直接按 `80 / ratio` 计算高度，当视频为 1:1 正方形 (ratio=1) 时，高度被计算为 80px，导致 Minimap 变成 80x80px 的巨大方块，在移动端横屏 short viewport 下被底部边界截断或挤出画面。
  * 现采用最大宽高上限约束策略（宽上限 80px，高上限 45px）：
    ```javascript
    const maxW = 80, maxH = 45;
    let miniW = maxW, miniH = Math.round(maxW / ratio);
    if (miniH > maxH) {
        miniH = maxH;
        miniW = Math.round(maxH * ratio);
    }
    ```
    对于 1:1 正方形视频，Minimap 尺寸自动约束为 45x45px；对于 16:9 视频约束为 80x45px，保证在任何屏幕尺寸和视频比例下均精准位于 `.tm-video-wrapper` 左下角 (`bottom: 20px; left: 20px;`)，绝不溢出截断。
* **零重排 GPU 加速与 100% 实时跟手 (Zero-Reflow GPU Acceleration)**：
  * **消除 `getBoundingClientRect()` 布局抖动**：以往在手势 `pointermove` 频率高达 120Hz 的事件处理器中，每次拖拽都强行调用 `video.getBoundingClientRect()` 触发 DOM 布局同步重排 (Layout Thrashing)，导致移动端拖拽极度卡顿、延时不跟手。
  * **预计算行程与 `transform: translateX` 替代 `left`**：在 `_updateConstraints()` 阶段提前计算并固化 Minimap Viewport 的最大行程 (`minimapMaxTravel = miniW - viewportW`)。在 `_applyOffset` 中直接映射计算 `miniX` 并通过 **`minimapViewport.style.transform = translateX(${miniX}px)`** 提交给 GPU 合成线程渲染，同时移除 CSS `transition` 延迟，实现 120fps 绝对零延迟丝滑跟手。
* **移动端横屏安全区与避让位移**：
  * 在 `_landscape_pc.css` 中将 Minimap 在横屏下的位置调整为 `bottom: calc(36px + env(safe-area-inset-bottom, 0px)) !important; left: calc(24px + env(safe-area-inset-left, 0px)) !important;`，完美抬高 16px 避开移动设备底部的 Home 条状按键与下方视口边缘遮挡，确保整个 Minimap 100% 完全呈现在视频区域内。

### 4.2 踩坑教训与避坑经验

#### 1. CSS Grid 规范中 `1fr` 隐藏的 `min-content` 撑高陷阱
* **坑点**：在 CSS Grid 布局中，习惯性给网格行设置 `grid-template-rows: 1fr`。但在 W3C CSS Grid 规范中，`1fr` 等价于 `minmax(auto, 1fr)`！当网格项（如评论边栏）内部包含非固定高度的 Flex 溢出内容时，`auto` 最小尺寸会以子元素的 `min-content` 高度（例如 524px）计算，直接导致 Grid 行的高度超越容器视口（430px），进而引发页面滚轴破版或底部固定栏溢出消失。
* **经验**：在需要严格限制 Grid 单元格高度等于视口/容器高度的场景下，**切勿直接使用 `1fr`**，必须显式书写为 **`grid-template-rows: minmax(0, 1fr)`**，明确告知浏览器允许该行高度收缩至 `0`。

#### 2. WebKit / Safari 引擎下多层 Flex 嵌套的 `height: 0` 必填法则
* **坑点**：仅给最外层 Flex 容器设置 `overflow: hidden` 与 `min-height: 0`，但在深层子级 Flex 元素（如列表项 Section、Section Body）中只写了 `flex: 1`。在 Blink/Gecko 上可能表现正常，但在 iOS WebKit 引擎中，未设 `height: 0` 的中间 Flex 子项仍会按内容实际高度展开，破坏最上层容器的 `flex-shrink` 计算。
* **经验**：在包含内部独立滚动（`overflow-y: auto`）的多层 Flex 布局中，所有参与高度分配的弹性子元素（自上而下每一级）都应当同时声明：
  ```css
  flex: 1 1 0% !important;
  height: 0 !important;
  min-height: 0 !important;
  ```
  结合底栏控件的 `flex-shrink: 0 !important;`，才能确保跨浏览器、跨平台表现 100% 一致。

#### 3. JS 内联 Style (`element.style.display`) 与 CSS 状态类 (`.is-swiping`) 冲突解耦
* **坑点**：以前在手势管理者（如 `videoSwipeManager`）中喜欢用 `element.style.display = 'block'` / `'none'` 强行控制 UI 控件（如 Minimap）的显隐。一旦 JS 写入内联 style，其优先级高于外部绝大多数 CSS Class 规则，导致后期想要通过 CSS 状态类（如 `body.controls-hidden` 或 `.is-swiping`）控制过渡动画与显隐逻辑时被内联样式锁死，产生难以调试的“样式失效”问题。
* **经验**：UI 控件的显隐与动画过渡应**尽可能通过 DOM 状态 Class (`classList.add/remove`) 进行表达**，避免在 JS 中直接操作 `style.display` / `style.opacity`。将视觉呈现完全退归 CSS 管理，代码逻辑将更具拓展性与可维护性。

### 5.5 踩坑教训与避坑经验

#### 1. 残留孤立代码块导致 Webpack 编译失败
- **坑点**：重构 fetchJavLibraryData 时，旧版 GM_xmlhttpRequest 调用片段（~15 行）残留在函数闭合括号后方，形成语法错误。文件体积大（1,890+ 行），肉眼 Review 未能发现，直到 npm run build 报错才定位。
- **教训**：大文件重构完成后必须通过 
pm run build 编译验证。养成"每阶段改完立即构建"的习惯。

#### 2. 多阶段重构应先建立工具层再改调用方
- **坑点**：直接进入 CommentScraper.js 修改时，发现每改一个 Fetcher 都要同时在 http.js 补充接口，思路频繁在两个文件间跳跃，容易漏改。
- **教训**：明确"工具层先行"原则：先建立稳定共享工具模块（Phase 3），再从上至下逐步替换调用方（Phase 4→5→6），单向依赖，路径可预测。

#### 3. 聚合微型文件时只做搬移，不混入逻辑优化
- **坑点**：合并 AdBlock 4 个微型类时，曾考虑顺手优化逻辑，导致 Diff 混乱（"是搬移还是修改？"）。
- **教训**：合并文件阶段只做"Cut + Paste"，行为不改变，验证通过后如需优化再单独提交，职责分离、提交单一。

#### 4. CSS 过渡动画期间不应切换 display — 正确的 Modal 显示序列
- **坑点**：旧版模态框 lement.style.display = 'block' + setTimeout(50ms) + opacity: 1，低端机上 rAF 时序不稳定，出现动画闪烁或直接跳跃到终态。
- **教训**：正确序列：先 display: flex，然后在同一 rAF 或 setTimeout(0) 后追加 .active Class 触发 CSS 过渡，让浏览器有机会在新 Frame 完成 Reflow 后启动过渡。

#### 5. Userscript 版本号的单一真相来源
- **坑点**：版本号仅存在于 webpack.config.js，package.json 无 version 字段，以往曾出现 meta 与 package.json 版本不一致的问题。
- **教训**：Userscript 项目版本号的单一真相来源应在 webpack.config.js 的 WebpackUserscript headers 中维护，由 Webpack 编译时自动注入到 dist/*.user.js 的 UserScript Header 注释中，无需手动修改 dist 产物。
